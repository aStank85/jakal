# CODEX_PATTERN_DETECTORS_INSIGHT_ENGINE_v1.md
## Goal
Add an Insight Engine that emits coaching-style “pattern” statements (not raw stats) using the scoped dataset and existing Team pairs analytics.

This layer answers:
- “When is this edge present?”
- “Where does it break?”
- “What’s a likely mechanism?”
- “What should I change?”

It does NOT invent causality. It produces **hypothesis-grade** insights with evidence and guardrails.

---

## Non-Negotiable Constraints
- No frontend build pipeline.
- No breaking API changes: add new endpoints/fields only.
- Must run fast: reuse scope cache; insight results must be cached.
- Must be deterministic and idempotent for a given scope.
- Must include “Evidence” metadata for each insight:
  - sample size, delta_pp, p_value/prob_above_baseline if available, volatility, scope info.
- Must avoid misleading language (“% more likely to lose”). Use:
  - “-3.1pp vs baseline” and “prob above baseline ~ 62%”.

# Architecture Overview

## A) Core concept
An **Insight** is a structured object:
- `id` stable string
- `title` short
- `message` natural language sentence(s)
- `severity`: INFO | WATCH | ACTION
- `category`: SYNERGY | ROLE | CONSISTENCY | MAP_POOL | TEMPO | RISK
- `entity`: pair/player/map/side/operator
- `evidence`: numbers + confidence/evidence tier
- `actions`: optional CTA objects (“Apply filter”, “Switch to Attack only”, etc.)

Insights are generated by a set of **Pattern Detectors**: pure functions that read a prepared feature bundle and emit 0..N insights.

## B) Where code goes
Create:
- `src/analytics/insights/`
  - `__init__.py`
  - `models.py` (Insight dataclasses + serialization)
  - `feature_builder.py` (builds reusable aggregates for detectors)
  - `detectors.py` (registry of detectors)
  - `engine.py` (runs detectors, ranks, caches)
  - `copy.py` (all user-facing sentences centralized)

Backend endpoint:
- `web/app.py`
  - `GET /api/workspace/insights/{username}`

Frontend:
- Add “Insights” view under Workspace (top row alongside Overview/Operators/Matchups/Team)
- Render insight cards with “Evidence” expanders and one-click actions to apply filters.

Caching:
- Add table `workspace_insights_cache` similar to team cache.
- Cache key includes `scope_key`, `insights_version`, `db_rev`.

# Feature Bundle (What Detectors Consume)

## Build once per scope (reuse scope cache match_ids)
Implement `build_insight_features(scope) -> dict` returning:

### Required:
1) `baseline`:
- `p0`, `wins`, `rounds` (scoped)
- if you have Wilson CI, include it

2) `pairs_overall`:
- from existing team endpoint computation (reuse module): list of pair stats
- include: matches_n, rounds_n, wins_n, win_rate, delta_pp, prob_above_baseline, p_value, volatility

3) `pairs_by_side`:
- same as pairs_overall but split by `side` (ATTACK/DEFENSE)
- Important: compute baseline_side too (user baseline on that side within scope)

4) `pairs_by_map`:
- compute top pairs per map (cap maps by total rounds)
- baseline per map too (user baseline on that map)

5) `player_side_profile` for the user:
From `player_rounds` + `round_outcomes` within scope:
- attack_rounds, defense_rounds
- attack_win_rate, defense_win_rate
- first_blood_rate (FB / rounds on attack)
- first_death_rate (FD / rounds on attack)
- “entry involvement” proxy:
  - `entry_rate = (first_blood + first_death)/atk_rounds`
- same for defense if meaningful

6) `pair_entry_effect` (lightweight, from existing data):
For each pair:
- user’s first_death_rate when paired vs when NOT paired (attack-only)
- user’s first_blood_rate when paired vs when NOT paired (attack-only)
This is feasible with current round-level flags and same-team detection.

### Performance rules
- Use scoped match_ids to restrict SQL.
- Use SQL aggregation wherever possible.
- Cap heavy expansions:
  - max pairs considered = top 80 by rounds
  - max maps considered = top 12 by rounds
- Everything must be cached.

# Detectors (v1 Set)

All detectors must:
- require minimum sample size
- include evidence fields
- emit ACTION only when effect is strong + stable
- phrase output as hypothesis:
  - “This suggests…”, “Likely…”, “May indicate…”

## Detector 1: Conditional Edge Collapse (Attack vs Defense)
Trigger:
- pair has strong positive overall: delta_pp >= +3 AND rounds>=120
- but Attack delta_pp <= 0 OR Attack prob_above_baseline < 0.55
Emit:
- Title: “Synergy is defense-skewed”
- Message:
  - “{A}+{B} is +X.Xpp overall, but on Attack it’s {atk_delta:+.1f}pp. This suggests the edge comes from defensive setups/holds, not executes.”
Actions:
- Apply filter: ws_side=defense
- Open Operators view

## Detector 2: “Good Pair” is Actually a Map One-Trick
Trigger:
- pair overall delta_pp >= +3 AND rounds>=120
- but >55% of pair rounds come from 1 map
Emit:
- “{A}+{B} edge is concentrated on {Map}”
Actions:
- Filter ws_map={Map}
- Toggle “All maps” comparison

## Detector 3: “Risk Pair” is consistent and should be avoided
Trigger:
- pair delta_pp <= -3 AND rounds>=120
- prob_above_baseline <= 0.35 (or p_value <= 0.05 and phat<p0)
- volatility low (<= 0.12) OR matches>=10
Emit ACTION:
- “Consistent underperformance with {A}+{B}”
Message:
- “Across {rounds} rounds, this pair is {delta}pp below baseline with low volatility. Treat as a comp risk unless roles change.”
Actions:
- Hide/flag pair (UI preference)
- Suggest “Try swapping entry/support roles”

## Detector 4: Volatile Edge (Looks good, isn’t stable)
Trigger:
- pair delta_pp >= +3 but volatility >= 0.20 and matches>=6
Emit WATCH:
- “Edge is volatile”
Message:
- “Win rate swings match-to-match. This may be opponent quality or inconsistent coordination.”
Actions:
- Narrow window to 30d
- Show rolling trend (future)

## Detector 5: Baseline Split Problem (You’re losing Attack / Defense disproportionately)
Trigger:
- abs(attack_win_rate - defense_win_rate) >= 8pp and each side rounds>=80
Emit ACTION:
- “Side imbalance”
Message:
- “Your Attack win rate is {atk_wr}% vs Defense {def_wr}%. Focus practice on the weaker phase.”
Actions:
- Filter ws_side=attack/defense
- Jump to Operators with side filter

## Detector 6: Pair reduces first deaths (Attack survivability)
Trigger:
- user first_death_rate when paired is lower by >= 3pp with rounds>=120
Emit ACTION:
- “{Pair} reduces your first deaths”
Message:
- “When queued with {B}, your first-death rate drops {X.X}pp on Attack. This suggests better spacing/trading or safer entry pacing.”
Actions:
- Filter to pair
- Show Attack-only

## Detector 7: Pair increases entry involvement but hurts win rate
Trigger:
- entry_rate increases >= +4pp with pair
- but delta_pp <= -2pp
Emit WATCH:
- “Over-aggression signal”
Message:
- “With {B}, you take earlier fights more often, but results worsen. Consider slowing pace or reassigning entry responsibility.”
Actions:
- Filter to Attack-only
- Suggest operator role shift (text only)

# Ranking & De-duplication

## Rank score
Each insight gets a rank score:
- base = abs(delta_pp) * sqrt(rounds)
- boost if evidence strong (prob far from 0.5)
- penalty if low sample or inconclusive
- ACTION severity can boost.

Dedupe:
- Don’t emit more than 1 insight per pair per category.
- If multiple detectors trigger on same pair, keep the one with higher rank score.

Limit:
- Return top 12 insights by rank.
